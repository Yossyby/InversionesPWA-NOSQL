# Data Model: Plataforma de Inversiones con IA (DR.FIC)

## Overview

This planning data model captures the minimum design entities implied by the canonical specification and the DR.FIC technical plan. It is intentionally architecture-level: it defines operational concepts and relationships without descending into implementation schemas.

## Entities

### User

- Purpose: Human actor who accesses dashboards, reviews recommendations, and explicitly approves or rejects assisted execution.
- Key fields:
  - `id` (UUID)
  - `auth_subject` (string)
  - `status` (`ACTIVE` | `INACTIVE`)
  - `role` (string)
  - `created_at` (timestamp)
- Validation rules:
  - Only active users can approve execution actions.
  - Authentication identity is derived from validated JWT claims.

### InvestmentAsset

- Purpose: Tradable asset tracked by the platform for analysis and monitoring.
- Key fields:
  - `id` (UUID)
  - `symbol` (string)
  - `asset_type` (`EQUITY` | `OPTION`)
  - `market` (`US`)
  - `status` (`ACTIVE` | `INACTIVE`)
- Validation rules:
  - v1 is limited to US equities and options.

### SignalRecord

- Purpose: Explainable signal candidate produced from active cores and available context.
- Key fields:
  - `id` (UUID)
  - `asset_id` (UUID, foreign key -> InvestmentAsset)
  - `signal_type` (`BUY` | `SELL` | `HOLD`)
  - `confidence_score` (decimal)
  - `status` (`GENERATED` | `REVIEWED` | `APPROVED_FOR_EXECUTION` | `REJECTED` | `EXPIRED`)
  - `created_at` (timestamp)
  - `expires_at` (timestamp, nullable)
- Validation rules:
  - A signal must remain explainable through linked evidence records.
  - AI evidence cannot be the sole support for a signal.

### CoreEvidence

- Purpose: Evidence generated by a single analytical core.
- Key fields:
  - `id` (UUID)
  - `signal_record_id` (UUID, foreign key -> SignalRecord)
  - `core_name` (string)
  - `verdict` (`BUY` | `SELL` | `HOLD` | `NEUTRAL`)
  - `rationale` (text)
  - `weight` (decimal)
  - `created_at` (timestamp)
- Validation rules:
  - Each evidence record must include an explainable rationale.
  - Core names must map to the constitutionally allowed analytical cores.

### RecommendationRecord

- Purpose: AI-assisted or deterministic recommendation derived from signals and contextual analysis.
- Key fields:
  - `id` (UUID)
  - `signal_record_id` (UUID, foreign key -> SignalRecord)
  - `summary` (text)
  - `risk_assessment` (text)
  - `source_type` (`DETERMINISTIC` | `AI_ENRICHED`)
  - `created_at` (timestamp)
- Validation rules:
  - Recommendations must reference a signal lifecycle record.
  - `AI_ENRICHED` recommendations cannot bypass human approval or deterministic evidence.

### BrokerAccount

- Purpose: Linked brokerage context used for portfolio sync and assisted execution.
- Key fields:
  - `id` (UUID)
  - `user_id` (UUID, foreign key -> User)
  - `broker` (`IBKR` | `ALPACA`)
  - `mode` (`LIVE` | `PAPER`)
  - `connection_status` (`CONNECTED` | `DISCONNECTED` | `ERROR`)
  - `last_sync_at` (timestamp, nullable)
- Validation rules:
  - Only IBKR and Alpaca are valid in v1.

### OrderIntent

- Purpose: Assisted execution request tied to a recommendation and broker account.
- Key fields:
  - `id` (UUID)
  - `recommendation_record_id` (UUID, foreign key -> RecommendationRecord)
  - `broker_account_id` (UUID, foreign key -> BrokerAccount)
  - `status` (`PENDING_APPROVAL` | `APPROVED` | `REJECTED` | `SUBMITTED` | `FAILED` | `COMPLETED`)
  - `approved_by_user_id` (UUID, nullable)
  - `approved_at` (timestamp, nullable)
  - `created_at` (timestamp)
- Validation rules:
  - No submission is valid without explicit manual approval.
  - A failed broker attempt returns the order flow to a new approval requirement before retry.

### ExecutionAttempt

- Purpose: Immutable log of each broker submission attempt.
- Key fields:
  - `id` (UUID)
  - `order_intent_id` (UUID, foreign key -> OrderIntent)
  - `broker` (`IBKR` | `ALPACA`)
  - `attempt_status` (`SUBMITTED` | `FAILED` | `FILLED` | `PARTIALLY_FILLED` | `CANCELLED`)
  - `error_code` (string, nullable)
  - `error_message` (text, nullable)
  - `attempted_at` (timestamp)
- Validation rules:
  - Failed attempts must preserve normalized error information.

### AuditEvent

- Purpose: Persistent trace of security, recommendation, approval, execution, and integration events.
- Key fields:
  - `id` (UUID)
  - `entity_type` (string)
  - `entity_id` (UUID)
  - `event_type` (string)
  - `actor_type` (`USER` | `SYSTEM` | `BROKER` | `AI`)
  - `payload` (JSON)
  - `occurred_at` (timestamp)
  - `retain_until` (date)
- Validation rules:
  - Retention is at least 365 days.
  - Security and execution actions always emit audit events.

### ContextDocument

- Purpose: Optional knowledge artifact used for AI retrieval augmentation and contextual reporting.
- Key fields:
  - `id` (UUID)
  - `source_type` (`DOCUMENT` | `MARKET_CONTEXT` | `HISTORICAL_REASONING`)
  - `reference` (string)
  - `embedding_ref` (string, nullable)
  - `created_at` (timestamp)
- Validation rules:
  - Optional in v1; presence cannot become a hard prerequisite for baseline signal generation.

## Relationships

- One `User` may own many `BrokerAccount` records.
- One `InvestmentAsset` may produce many `SignalRecord` records over time.
- One `SignalRecord` has many `CoreEvidence` records and zero or many `RecommendationRecord` records.
- One `RecommendationRecord` may lead to many `OrderIntent` records across brokers or retries.
- One `OrderIntent` has many `ExecutionAttempt` records.
- Any major domain entity may emit many `AuditEvent` records.
- Optional `ContextDocument` records may support many AI-assisted recommendation flows.

## State Transitions

### SignalRecord

- `GENERATED` -> `REVIEWED`
- `REVIEWED` -> `APPROVED_FOR_EXECUTION`
- `REVIEWED` -> `REJECTED`
- `REVIEWED` -> `EXPIRED`

### OrderIntent

- `PENDING_APPROVAL` -> `APPROVED`
- `PENDING_APPROVAL` -> `REJECTED`
- `APPROVED` -> `SUBMITTED`
- `SUBMITTED` -> `COMPLETED`
- `SUBMITTED` -> `FAILED`
- `FAILED` -> `PENDING_APPROVAL`

## Persistence Notes

- Operational entities belong in Supabase.
- Historical reasoning, large context payloads, and optional retrieval data may live in MongoDB or a vector-capable supporting store.
- Persistence remains strictly server-side.
